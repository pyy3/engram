# Engram Memory Schema

This file defines what each memory layer should contain, how files should be
structured, and governance rules for the LLM maintaining the knowledge base.

## Layer Definitions

### semantic/ — Facts & Patterns

**Purpose**: Distilled knowledge about how things work.

**Each file should contain**:
- A clear `# Title` that names the concept/entity/pattern
- Factual observations (what was tested and confirmed)
- Gotchas and edge cases discovered empirically
- Examples with exact values/commands that worked

**Naming**: `{topic}.md` or `{entity}-{aspect}.md`
- Good: `auth-jwt-patterns.md`, `database-migration-gotchas.md`
- Bad: `notes.md`, `stuff-i-learned.md`, `2026-03-21.md`

**Do NOT put here**: Step-by-step instructions (→ procedural/), today's progress (→ active/), raw session logs (→ lossless/)

---

### procedural/ — Workflows & How-Tos

**Purpose**: Repeatable processes documented as step-by-step guides.

**Each file should contain**:
- A clear `# Title` in imperative form ("Deploy to Production", "Set Up Auth")
- Numbered steps (1, 2, 3...)
- Exact commands with copy-pasteable code blocks
- Prerequisites listed at the top
- Known failure modes and recovery steps

**Naming**: `{action}-{target}.md` or `{workflow-name}.md`
- Good: `deploy-production.md`, `setup-local-dev.md`, `debug-memory-leaks.md`
- Bad: `how-to.md`, `process.md`

**Do NOT put here**: Reference facts (→ semantic/), one-time task lists (→ active/)

---

### active/ — Today's Working Context

**Purpose**: Mutable session state. What you're doing right now.

**Files**:
- `current-tasks.md` — What's done, in progress, and blocked TODAY
- `context-inject.md` — Auto-populated by `engram start` (don't edit manually)
- Other transient notes specific to current work session

**Rules**:
- Overwritten each session (not permanent)
- Update `current-tasks.md` every 3-4 tool calls
- Move durable insights to semantic/ or procedural/ before they're lost

---

### lossless/ — Session History

**Purpose**: Immutable record of what happened. Audit trail.

**Files**: `YYYY-MM-DD_HHMM.md` (auto-generated by `engram end`)

**Rules**:
- Never edit after creation
- Never delete
- Created every ~30 minutes during active work
- Contains: summary, tasks snapshot, files changed

---

## Governance Rules

1. **Never delete** — supersede with newer files or append "[SUPERSEDED]" to title
2. **One topic per file** — if a file covers 2+ unrelated topics, split it
3. **Date your claims** — if something is time-sensitive, include "as of YYYY-MM-DD"
4. **Show your work** — include the command/test that proved a fact
5. **Flag uncertainty** — prefix unverified claims with "UNVERIFIED:" or "HYPOTHESIS:"
6. **Cross-reference** — run `engram xref` periodically to link related files
7. **Lint regularly** — run `engram lint` to find stale/orphan content
8. **Compound discoveries** — run `engram compound "topic"` when you notice scattered knowledge

## Anti-Patterns

- Files named "misc", "notes", "random", "temp" — be specific
- Duplicating content across files — cross-reference instead
- Mixing facts and instructions in one file — split into semantic + procedural
- Leaving stale "TODO" or "in progress" in semantic/ — that belongs in active/
- Writing session-specific details in semantic/ — use lossless/ for raw history
